# 제품 소프트웨어 저작권

## 제품 소프트웨어 저작권 보호

### DRM(Digital Rights Management)
- 디지털 콘텐츠의 생성에서부터 실제 사용자까지 모든 유통 과정에 걸쳐 콘텐츠를 안전하게 관리 및 보호하고 허가된 사용자만이 접근할 수 있도록 제한

### 특설
- 거래 투명성
- 사용 규칙 제공
- 자유로운 상거래 제공

### DRM 기술 요소
- 암호화
- 암호화 키 관리
- 암호화 파일 생성
- 식별 기술
- 저작권 표현
- 정책 관리
- 크랙 방지
- 인증
- 인터페이스
- 이벤트 보고

### DRM 용어
- 콘텐츠 제공자
- 콘텐츠 분배자: 쇼핑몰 등으로 암호화된 콘텐츠 제공
- 패키저: 콘텐츠를 메타 데이터와 함께 배포 가능한 단위로 묶는 기능
- 보안 컨테이너: 원본을 안전하게 유통하려는 전자적 보안 장치
- DRM Controller: 배포된 콘텐츠의 이용 권한을 통제
- Clearing House: 키 관리 및 라이선스 발급 관리
    - 저작권료의 정산과 분배가 이루어지는 곳

### 디지털 콘텐츠의 사용 권한 유형
- 렌더 퍼미션
    - 사용자에게 **콘텐츠가 표현**되고 이용되는 권리 형태를 정의
    - 문서, 동영상
- 트랜스포트 퍼미션
    - 사용자들 간에 **권리 교환**이 이루어지는 권리 형태를 정의
    - 카피, 무브, 론
- 테리버티브 퍼미션
    - 콘텐츠의 **추출 변형**이 가능한 권한

# 제품 소프트웨어 매뉴얼 작성

## 소프트웨어 메뉴얼

### 소프트웨어 메뉴얼
- 제품 소프트웨어 개바 단계부터 적용한 기준이나 패키징 이후 설치와 사용자 측면의 주요 내용 등을 기록한 문서
- 설치 매뉴얼과 사용자 매뉴얼로 구분됨

### 설치 메뉴얼
- 소개, 설치 파일, 설치 절차 등이 포함
- 표시될 수 있는 예외 상황에 관련된 내용을 별도로 구분하여 설명
- 완료할 때까지의 전 과정을 빠짐없이 순서대로 설명
- **사용자 기준**으로 작성
- 목차, 개요, 기본 사항 등이 기본적으로 포함

### 메뉴얼 구성
- 목차 및 개요
- 문서 이력 정보
- 설치 매뉴얼 주석
- 설치 도구의 구성
- 설치 위치 지정

### 소프트웨어 설치 매뉴얼 기본 사항
- 제품 소프트웨어 개요
- 설치 관련 파일
- 설치 아이콘
- **프로그램 삭제**
- 관련 추가 정보

### 사용자 메뉴얼 작성 프로세스
- 작성 **지침 정의**
- 구성 요소 정의
- 구성 요소별 내용 작성
- 사용자 메뉴얼 **검토**

### 소프트웨어 국제 표준 품질 특성
- ISO/IEC 9126
    - 소프트웨어 품질 특성과 척도에 관한 지짐

- ISO/IEC 12119
    - 일반적인 제품 품질 요구사항 및 **테스트를 위한 국제 표준**
    - 제품 설명서, 사용자 문서 및 프로그램으로 구분

- ISO 9001
    - 설계, 개발, 생산, 설치 및 서비스 과정에 대한 품질 보증 모델

### 소프트웨어 품질 목표
- 운영 특성
    - 정확성
    - 신뢰성
    - 사용 용이성
    - 효율성
    - 무결성

- 변경 수용 특성
    - 이식성
    - 상호 운용성
    - 재사용성

- 변경 수용 특성
    - 유지보수성
    - 유연성
    - 시험 역량

### 소프트웨어 품질 측정 시 관점별 분류
- 사용자 관점
    - 신뢰성, 효율성, 사용 용이성, 간결성

- 개발자 관점
    - 검증 가능성, 유지보수성, 이식성, 무결성, 사용성

- 프로젝트 관리자 관점
    - **프로세스 생산성과 제어 용이성**

# 형상 관리

## 형상 관리 도구

### 형상 관리
- 개발 단계에서 생성되는 모든 문서, 코드 등 소프트웨어의 변경 사항을 체계적으로 관리하기 위하여 추적하고 통제하는 것
- 형상 항목이라는 형태로 선정하고, 형상 항목 간의 변경 사항 추적과 통제 정책을 수립하고 관리함
- 요구사항 변경 또는 오류로 지속해서 변화하는 자료이며 이러한 변화를 이력화하여 유지보수성을 최신화

### 형상 관리 항목
- 프로젝트 요구 분석서
- 운영 및 설치 지침서
- 요구사항 명세서
- 설계/인터페이스 명세서
- 테스트 설계서
- 소프트웨어 품질 보증
- 형상 관리
- V&V 계획서와 같은 계획서
- 코드 모듈

### 형상 관리 절차
- 형상 식별
- 형상 통제
    - 기준선을 기반으로 뭔가를 할거야
- 형상 보고 및 감사
- 형사 기록 보고

### 형상 관리, 버전 관리, 변경 관리
- 형상 관리 > 버전 관리 > 변경 관리

### 형상 관리 역할
- 프로젝트 개발 비용을 효율적으로 관리하는 것은 아님(비용을 줄일 수는 있는 것)

## 버전 관리

### 도구 구분
- 공유 폴더 방식
- 클라이언트 서버 방식
    - CVS, SVN
- 분산 저장소 방식
    - Git
    - 원격 / 로컬 저장소가 함께 저장되어 관리됨

### 주요 버전 관리 도구
- CVS(Concurrent Versions System)
    - 동시 버전 시스템
    - 파일로 이뤄진 모든 작업과 모든 변화를 추적하고 여러 개발자가 협력하여 작업할 수 있게 함

- RCS
    - CVS와의 차이점은 수정을 한 사람만으로 제한

- SVN (서버 클라이언트 방식 - 공유 폴더 방식 X)
    - CVS 보다 속도 개선

### git 주요 명령어
- init: 새로운 로컬 git 생성하기
- add: 저장소에 파일을 추가하기
- commit: 작업 내역 지역 저장소 저장하기
- branch: 새로운 파생 저장소인 브랜치 생성하기
- checkout: 선택한 브랜치로 이동하기
- merge: 현배 브랜치와 지정한 브랜치를 병합하기
- fetch: git 서버에서 코드를 받아오기
- pull: git 서버에서 최신 코드 받아와 병합하기
- remote: 원격 저장소 추가하기
- clone: 원격 저장소에 있는 프로젝트 복사하여 내려받기

### Subversion(SVN) 주요 명령어
- import: 아무것도 없는 서버의 저장소에 맨 처음 소스 파일을 저장
- check-in: 체크 아웃으로 가져온 파일을 수정 후 저장소에 새로운 버전으로 갱신
- check-out: 타 개발자가 수정 작업을 위하여 저장소에 저장된 파일을 자신의 작업 공간으로 인출
- commit: 체크인 시 이전 갱신 사항이 있는 경우 충돌이 있을 경우 알림을 표시하고 diff 도구를 이용하여 수정한 뒤 commit 과정을 수행
- diff: 새로운 개발자가 추가된 파일의 수정 기록을 보면서 기존 개발자 변경 사항 추적
- update: 저장소에 존재하는 최신 버전 자료와 자신의 작업 공간과 동기화
- branch: 주 저장소에서 파생된 프로젝트
- fork: 소스 코드를 통째로 복사하여 독립적인 새로운 소프트웨어 개발 허용
- update: commit 후 새로운 개발자가 자신의 작업 공간과 저장소를 동기화
- info: 정보
- merge: 병합

# 애플리케이션 테스트 관리

## 테스트 케이스

### 소프트웨어 테스트
- 품질 향상 
- 오류 발견 관점
- 오류 예방 관점
    - 동료 리뷰
    - 인스펙션을 통해서 오류를 사전에 예방

- 테스트 케이스도 계속해서 발전해야함

### 파레토의 법칙
- 80대 20법칙
- 전체 결과의 80%가 전체 원인의 20%에서 일어나는 현상을 가리킴

### 테스트 오라클
- 테스트의 결과가 참인지 거짓인지를 판단하기 위해서 사전에 정의된 참값을 입력하여 비교하여 기법 및 활동을 말함

1. 참 오라클
    - 모든 입력값에 대하여 적합한 결과를 생성하고 발생한 오류를 모두 검출할 수 있는 오라클
2. 일관성 검사
    - 애플리케이션 변경이 있을 때, 수행 전과 후의 결과값이 동일한지 확인하는 오라클
3. 샘플링 오라클
    - 임의로 선정한 몇 개의 입력값에 대해서만 기대하는 결과를 제공
4. 휴리스틱 오라클
    - 휴치스틱(추정)으로 처리

### V 모델
- 단위 테스트
    - 통합 테스트
        - 시스템 테스트
            - 인수 테스트(고객이 테스트)
- 이건 동적

### 시각에 따른 테스트
- 검증(Verification) 테스트
    - 제품이 명세서대로 완성되었는지 검증하는 단계
    - **개발자++의 시각에서 제품의 생산 과정을 테스트하는 것을 의미함

- 확인(Validation) 테스트
    - 사용자의 요구사항을 잘 수행하고 있는지 확인
    - **사용자의 시각**에서 생산된 제품의 결과를 테스트하는 것을 의미함

### 테스트 케이스 자동 생성
- 자료 흐름도
    - 테스트 경로 관리, 입력 도메인 분석
        - 테스트 데이터 산출, 랜덤 테스트
            - 무작위 값 입력, 신뢰성 검사

### 테스트 레벨의 종류
- 단위 테스트
    - 개발자가 원시 코드를 대상으로 각각의 단위를 다른 부분과 연계되는 부분은 고려하지 않고 단위 자체에만 집중하여 테스트
    - 객체지향에서 클래스 테스팅

- 통합 테스트
    - 컴포넌트간 인터페이스 및 연동 기능 등으 구조적으로 접근하여 테스트

- 시스템 테스트
    - 단위/통합 테스트가 가능한 완벽히 완료되어 기능상에 문제가 없는 상태에서 실제 환경과 가능한 유사한 환경에서 진행

### 알파 테스트와 베타 테스트
- 알파 테스트 (내부에서 한다고 이해)
    - 개발자 관점에서 수행
    - 사용상의 문제를 반영되도록 하는 테스트
    - 개발자의 장소에서 사용자가 개발자 앞에서 행함
    - 개발자는 사용상의 문제를 기록하여 반영되도록 하는 테스트

- 베타 테스트 (외부에서 한다고 이해)
    - 선정된 다수의 사용자가 자신들의 환경에서 테스트

# 테스트 시나리오와 테스트

## 테스트 기법

### 화이트 박스 테스트
- 모듈의 원시 코드를 오픈 시킨 상태에서 코드의 논리적인 모든 경로를 테스트하는 방법
- 소스 코드의 모든 문장을 한 번 이상 수행하여 모듈 안의 작동을 직접 관찰할 수 있음
- 논리적 경로를 검사

- 종류
    - 기초 경로 검사(제어 흐름)
    - 제어 구조 검사
        - 조건 검사
        - 루프 검사
        - 데이터 흐름 검사

### 화이트 박스 테스트 검증 기준
- 문장 검증 기준
    - 모든 구문이 한 번 이상 수행
- 분기 검증 기준
    - 소스 코드의 모든 조건문이 한 번 이상 수행
- 조건 검증 기준
    - 소스 코드의 모든 조건문에 대해 조건이 True인 경우와 False인 경우가 한 번 이상 수행

### 블랙 박스 테스트 종류
- 동치 분할 검사
- 원인-효과 그래프 검사
- 오류 예측 검사
- 비교 검사: 여러 버전의 프로그램에 동일한 검사 자료를 제공하여 동일한 결과가 출력되는 검사하는 기법
- 경계값 분석

## 순환 복잡도

### 순환 복잡도(Cyclomatic)
- 제어 흐름 난이도의 복잡도에 따라 프로그램의 이해 난이도가 결정됨
- 복잡도를 싸이클로메틱 개수에 의해서 산정하는 방법
- 최대 10을 넘지 않도록 하며 넘으면 이를 분해하도록 함

### 계산 방식
- 복잡도: 화살표 수 - 노드 수 + 2 

## 해싱

### 해싱 함수의 종류
- 제산 방법
    - 나머지 연산자를 사용해서 테이블 주소를 계산하는 방법

- 중간 제곱 방법(Mid-Square Method)
    - 레코드 키값을 제곱한 후에 결과값의 중간 부분에 있는 몇 비트를 선택하여 해시 테이블의 홈주소로 사용

- 중첩 방법(폴딩법)
    - 나눈 부분의 각 숫자를 더하거나 XOR 한 값을 홈주소로 사용

- 기수 변환 방법(Radix Conversion Method)
    - 레코드 키를 구성하는 수들이 모든 키들 내에서 자리별로 어떤 **분포**인지 조사

- 무작위 방법(Random Method)
    - 난수를 발생시킨 후

- 숫자 분석법(Digit Analysis)

- 동의어
    - 해싱에서 동일한 홈주소로 인하여 충돌이 일어난 레코드들의 집합을 의미

## 모듈 연계

### EAI 유형 (Enterprice Application Integration)
- point to point
    - 애플리케이션을 중간 미들웨어 없이 포인트 투 포인트로 연결하는 방식

- Hub & Spoke
    - 단일 접점인 허브 시스템을 통해 데이터를 전송하는 중앙 집중형 방식
    - 확장 및 유지보수가 수월
    - 중앙 허브에 문제 발생 시 모든 시스템에 영향을 줌

- Message Bus
    - 애플리케이션 사이에 미들웨어를 배치하여 처리하는 방식으로 확장성이 뛰어남
    - 대용량 데이터 처리에 유리

- Hybrid
    - Hub & Spoke와 Message Bus의 혼합 방식
