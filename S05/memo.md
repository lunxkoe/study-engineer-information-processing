# 소프트웨어 개발 방법론

## 소프트웨어 개발 방법론의 개요

### 구조적 방법론
- 정형화된 분석 절차에 따라 사용자 요구사항을 파악하여 문서화하는 체계적인 방법론
- 요구사항 분석, 구조적 분석, 구조적 설계, 구조적 프로그래밍 단계로 구성
- DFD를 주로 사용함
- 설계 구조도 사용

### 객체지향 방법론
- 요소
    - 객체, 클래스, 메시지

- 원칙
    - 캡슐화
    - 정보 은닉
    - 추상화
    - 상속
    - 다형성

### 컴포넌트 기발 개발 방법론(CBD)
- 재사용 사능한 컴포넌트의 개발 또는 상용 컴포넌트들을 조합하여 애플리케이션 개발 생산성과 품질을 높이고, 시스템 유지보수 비용을 최소화

### CBD SW 개발 표준 산출물
- 분석
    - 사용자 요구사항 정의서, 유스케이스 명세서, 요구사항 추적표

- 설계
    - 클래스 명세서
    - 사용자 인터페이스 설계서
    - 아키텍처 설계서
    - 총괄 시험 계획서

- 구현

- 시험

### CBD 특징
- 개발 준비, 분석, 설계, 구현, 테스트, 전개, 인도 순으로 반복 점진적 개발 프로세스를 제공
- 컴포넌트 설계서
- 개발 기간 단축
- 생산성 향상
- 새로운 기능 추가가 쉬워짐

### 합성 중심과 생성 중심
- 합성 중심
    - 블록 합치기

- 생성
    - 추상화 형태로 쓰여진 명세를 구체화하여 프로그램을 만드는 방법
    - 패턴 구성 방법

# 비용 산정 모델

### 비용 산정 모델 종류
- 전문가 감정 기법
- 델파이 기법
- LOC(Line Of Code) 기법
- COCOMO(COnstructive COst MOdel) 모델
- Putnam 모델
- 기능 점수(FP: Functional Point)

### 델파이 기법
- 산정 요원과 조정자에 의해 산정하는 기법
- **전문가가 독자적으로 감정할 때 발생할 수 있는 편차를 줄이기 위한 목적**
- 의견 일치가 이루어지지 않을 경우 의견을 근거로 익명으로 집단 내에 배포하고 자신들의 산정을 수정할 수 있도록 함

### LOC 기법
- 각 기능의 원시 코드 라인 수의 비관치, 낙관치, 기대치를 측정하여 예측치를 구하고, 이를 이용하여 비용을 산정하는 기법
- 예측치 = (a + (4 * c) + b) / 6
    - a: 낙관치
    - b는 비관치
    - c는 기대치

- 문제
    - 라인 수 36000라인
    - 프로그래머 6명
    - 평균 생산성이 월간 300라인
    - 소요 기간
    - 36000 / (6 * 300) = 20개월

### COCOMO 모델
- 보헴
- 같은 규모의 소프트웨어라도 유형에 따라 비용이 다르게 산정됨
- 비용 산정 수식을 이용
- 특정을 15개로 나누고 각각에 대한 승수 값을 제시
- 개발 유형
    - 단순형(Organic Mode): 5만 라인 이하의 소프트웨어를 개발하는 유형
    - 중간형(Semi-detached): 30만 라인이하의 소프트웨어를 개발
        - 트랜잭션 처리 시스테이나 운영체제, 데이터베이스 관리 시스템

    - 임베디드형: 30만 라인 이상의 소프트웨어를 개발하는 유형
        - 초대형 규모의 트랜잭션 처리 시스템, 운영체제 등

### Putnam
- Rayleigh-Norden 곡선의 노력 분포도를 이용한 프로젝트 비용 산정 기법
- 개발 생명주기의 전 과정 동안에 사용될 노력의 분포를 예측
- **SLIM**: Rayleigh-Norden 곡선과 Putnam의 모형에 기반을 둔 자동화 추정 도구

### FP
- 기능마다 가중치를 줌 
- 비용 산정 요소
    - 코드 라인 수, 데이터 파일 수, 문서 페이지 수, 입력 유형의 수, 출력 보고서의 수, 외부 루틴과의 인터페이스 수, 명령어(사용자 질의 수)

5. 최적 단계

# 네트워크 구성

## 네트워크 구성

### 성형(Star Topology)
- 중계기 중심으로 여러 단말기가 붙어있는 형태(모든 단말기가 연결되있지 않음)

### 링형
- 단말기간 연결되어 있는 형태

### 버스
- 버스 정류장 형태

### 망형(링형과 다름)
- 단말기가 모두 연결되어있는 상태
- n(n-1) / 2개

## 네트워크 장비

### 허브
- 한 포트에서 수신된 신호를 다른 모든 포트로 재전송하는 장치

### 브리지
- 데이터 링크 계층에서 동작
- 같은 MAC 프로토콜을 사용하는 근거리 통신망 사이를 연결하는 통신 장치

### 스위치
- 브리지와 같이 두 개 이상의 LAN을 연결하여 하나의 네트워크로 만드는 장치
- 보안 및 트래픽 관 기능도 할 수 있음
- 스위치 분류
    - 각 숫자에 해당하는 계층을 담당함
    - L2 
    - L3
    - L4
    - L7

### 라우터
- 경로 설정

### 게이트웨이
- 네트워크를 식별하기 위한 기점

### VLAN
- 물리적 배치와 상관없이 논리적으로 LAN을 구성하여 Broadcast Domain을 구분할 수 있게 해주는 기술

- 벨만 포드 알고리즘 사용

- IGP(Interior Gateway Protocol)

### MQTT
- IBM
- IOT
- TCP/IP 기반 네트워크에서 동작하는 발행 - 구독 기반의 메시징 프로토콜로 최근 IOT에 사용함

- 전력망을 스마트하게 관리하는 것

- **기존 무선 랜의 한계를 극복하기 위해 등장**
- 대규모 디바이스 네트워크 생성에 최적화되어 차세대 이동 통신, 홈네트워킹, 공공 안전 등의 특수 목적을 위한 새로운 방식의 네트워크 기술

### Wavelength Division Multiplexing(파장 분할 다중화)
- 래이저 빛을 사용함
- 광신호를 동시에 이용하는 것(파장이 서로 다른)

# 보안 방법론

### 소프트웨어 개발 보안 방법론의 개념
- 요구사항에서부터 보안 항목을 식별

### Seven TouchPoints
- 소프트웨어 보안의 모법 사례를 SDLC에 통합한 개발 보안 방법론
- 공통 위험 요소를 파악하고 이해하며, 보안을 설계하고 소프트웨어 산출물에 대해 철저하고 객관적인 위험 분석 및 테스트를 거쳐 안전한 소프트웨어를 만들어내는 방법을 정의하고 있음

### 정보보안의 3대 요소
- 기밀성
    - 인가된 사용자만 정보 자산에 **접근할 수 있음**
    - 방화벽, 암호 패스워드
- 무결성
    - 오직 인가된 사용자가 인가된 방법으로 정보를 **수정할 수 있음**
- 가용성
    - 사용자가 필요할 때 데이터에 접근할 수 있는 능력을 말함
    - 데이터 백업, 위협 요소 제거
    - 서비스 거부 등과 같은 공격에 의해 위협받을 수 있음

### OWASP
- The Open Web Application Security Project
- 오픈소스 웹 애플리케이션 보안 프로젝트로서 주로 웹을 통한 정보 유출, 악성 파일 및 스크립트, 보안 취약점 등을 연구하는 곳

# 보안 솔루션과 아키텍처

## 보안 솔루션

### IDS
- 침입 탐지 시스템
- HIDS(호스트 기반)
    - 컴퓨터 시스템 내부를 감시하고 분석하는 기술

- NIDS(네트워크 기반)
    - 네트워크 상의 패킷을 분석

### 침입 탐지 기법
- 오용 탐지
    - 이미 알려진 공격 패턴과 일치하는지를 검사하여 침입을 탐지함
    - False Positive가 낮은 반면 False Negative가 높음
    - 속도가 빠르고 구현이 간단

- 이상 탐지
    - 장기간 수집된 사용자 행동 패턴을 활용해 통계적으로 침입을 탐지


### IPS
- 침입 방지 시스템

### IPSec
- 암호화 수행 시 양방향 암호화 지원

### ESM 
- 통합 탐지 시스템
