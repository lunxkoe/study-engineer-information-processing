- Fan-in: 상위 모듈 수: 해당 모듈로 들어가고 있는 아이들
- Fan-out: 하위 모듈 수: 해당 모듈에서 나가고 있는 아이들
- **기점 찾기(모듈)**
- Depth: 상위 모듈에서 주어진 모듈까지의 깊이
- Width: 같은 등급의 모듈 수

- 복잡도 최적화를 위한 조건
    - Fan-in은 높이고
    - Fan-out은 낮추고

### 순차 코드
- 코드화 대상 항목을 어떤 일정한 배열로 일련번호를 배당하는 코드로 확장성이 좋으며, 단순해서 이해하기 쉽고, 기억하기 쉬움
- 항목 수가 적고, 변경이 적은 자료에 적합하며, 일정 순서대로 코드를 할당하므로 기억 공간 낭비가 적음
- 누락된 번호를 삽입하기 어력보 명확한 분류 기준이 없어 코드에 따라 분류가 어려워 융통성이 낮음

### 블록 코드
- 블록 안에서 일련번호를 배정하는 코드
- 기계처리가 어려움

### 그룹 분류식 코드
- 대분류, 중분류, 소분류

### 10진 분류 코드
- 100 | 국문학
- 200 | 철학
- 300 | 정보학
- 이런 느낌

### 표의 숫자 코드(Significant Digit Code, 유효 숫자 코드)
- 코드화 대상 항목의 길이, 넓이, 부피, 무게 등을 나타내는 문자나 숫자, 기호를 그대로 코드로 사용
- 예: 127-890-1245: 두께: 127mm, 폭 890mm

### 연상 코드
- TV-39-U | TV 39인치 컬러

### 코드 올류 종류
- 필사 오류: 입력시 한 자리를 잘못 기록하는 오류 1234 - 1237 (Transcription Error)
- 전위 오류: 입력 시 좌우 자리를 바꾸어 발생하는 오류 1234 - 1243 (Transposition Error)
- 이중 오류: 전위 오류가 두 개 이상 발생 1234 - 2143
- 생략 오류: 입력 시 한 자리를 추가해서 기록하는 오류 1234 - 123
- 추가 오류: 입력 시 한 자리를 추가해서 기록하는 오류 1234 - 12345
- 임의 오류: 두 가지 이상의 오류가 결합해서 발생하는 오류 1234 - 21345 (Random Error)

### 구조적 분석

### 구조적 분석 도구 - 자료 흐름도(DFD)
- 입력, 출력 자효 자체에 대해서만 인지하고 자료의 위치나 방향은 알 필요가 없음

### 데이터(자료) 흐름도 - 표기법 암기
- 구성요소 / 의미 / 표기법
- 프로세스
- 자료 흐름
- 자료 저장소
- 단말(Terminator)

### 소단위 명세서 (Mini Specification)
- **세분화**된 자료 흐름도에서 최하위 단계 버블(프로세스)의 처리 절차를 기술한 것으로 프로세스 명세서라고도 함

### 자료 사전(DD)
- 기호 / 의미 / 설명
- = / 자료의 정의 / ~로 구성되어 있다
- + / 자료의 연결 / 그리고
- () / 자료의 생략 / 생략 가능한 자료
- [|] / 자료의 선택 / 다중 택일, 또는
- {} / 자료의 반복 / n이 아래(최소 N번 이상 반복), n이 위에(최대 N번 이하 반복), 위n/아래m: m번 이상, n번이하

### 자료 사전의 역할
- 자료 흐름도에 기술한 모든 자료의 정의를 기술할 문서
- 구조적 시스템 방법론에서 자료 흐름도, 소단위 명세서와 더불어 중요한 문서

### 자료 사전에 기술해야할 자료
- 자료 항목
- 자료 저장소
- 자료에 대한 의미
- 자료 원소의 단위 및 값

### 모듈
- 전체 프로그램에서 어떠한 기능을 수행할 수 있는 실행 코드를 의미함
- **재사용성**
- **자체적으로 컴파일 가능**
- 노동력 및 개발 비용 절감
- 모듈의 독립성
    - 결합도
    - 응집도
- 모듈마다 사용할 변수를 정의하지 않고 상속하여 사용할 수 있음

### 결합도
- 두 모듈 간의 상호 의존도로서 두 모듈 간의 기능적인 연관 정도를 나타냄
- **낮을 수록 좋음**

- 결합도 약함 순
    - 자료(데이터) 결합도 - 인터페이스로 통신하는 것
    - 스탬프 결합도 - 자료구조
    - 제어 결합도
    - 외부 결합도
    - 공통 결합도
    - 내용 결합도
    - **자스제외공네**

### 응집도
- 모듈 안의 요소들이 서로 관련된 정도를 말함
- 독립적 기능으로 구성된의 정도를 의미함
- **높을수록 좋음**

- 응집도 강함 순
    - 기능적 응집도
    - 순차적 응집도
    - 교환적 응집도
    - 절차적 응집도
    - 시간적 응집도
    - 논리적 응집도
    - 우연적 응집도
    - **기순교절시논우**

### 모듈의 설계의 특징
- 응집도는 높게 / 결합도는 낮게
- 유지보수 수월, 복잡도와 중복을 피하고 입구와 출구는 하나씩 갖도록 함
- 계층적

### 모듈과 컴포넌트 차이
- 컴포넌트는 인터페이스를 가짐

### 재사용 규모에 따른 구분
- 함수와 객체: 클래스, 메소드 단위로 소스 코드 등을 재사용함
- 애플리케이션: 공통 업무를 처리할 수 있도록 구현된 애플리케이션을 공유하여 재사용
- 컴포넌트: 컴포넌트 자체 수정 없이 인터페이스를 통하여 컴포넌트 단위로 재사용

### SOLID
- 단일 책임의 원칙: SRP: 모든 클래스는 단일 목적으로 생성되고 하나의 책임만 가져야함
- 개방-폐쇄의 원칙: OCP: 확장에는 열려있고, 변경에는 닫혀있어야한다
- 리스코프치환 원칙: LSP: 부모 클래스가 들어갈 자리에 자식 클래스를 대체하여도 계획대로 동작해야한다 (상속)
- 인터페이스 분리 원칙: ISP: 클라이언트는 자신이 사용하지 않는 메소드와 의존 관계를 맺으면 안된다
- 의존 역전 원칙: DIP: 의존 관계를 맺으면 변하기 쉽고 변화 빈도가 높은 것보다 변하기 어렵고 변화 빈도가 낮은 것에 의존한다

2. 브릿지 패턴
3. 컴포시트 패턴
4. 데코레이터 패턴
5. 퍼사드 패턴

### 행위 패턴
- 반복적으로 사용되는 객체들의 상호작용을 패턴화 한 것
- 클래스나 객체들이 상호작용하는 방법과 책임을 분산하는 방법을 정의
1. 책임 연쇄
2. 반복자
3. 명령
4. Interpreter
5. 감시자(Observer)
6. 기록(Memento)
7. 전략
8. 중제자
    - 객체간의 통제와 지시의 역할을 하는 중재자를 두어 객체 지향의 목표를 달성하게 함

### 디자인 패턴 vs 아키텍처 패턴
- 아키텍처 패턴
    - 시스템 전체 구조를 설계

- 디자인 패턴
    - 서브 시스템 내 컴포넌트와 그들 간의 관계를 구성

### 미들웨어 솔루션의 정의
- 클라이언트와 서버 간의 통신을 담당하는 시스템 소프트웨어

### 미들웨어 솔루션의 유형
- 데이터베이스
- TP-Monitor(Transaction Processing - Processing Monitoring)
    - 트랜잭션이 일어나는 것을 모니터링 하는 것
- ORB(Object Request Broker)
- RPC(Remote Procedure Call)
- MOM(Message Oriented Middleware)
- WAS(Web Application Server)
- 객체 트랜잭션 모니터(ORB + TP-Monitor)

### 미들웨어 솔루션 분류
- DB 미들웨어 (애플리케이션 - TO - 데이터 방식)
    - ODBC
    - IDAP
    - DRDA
    - OLEDB

- 통신 미들웨어 (애플리케이션 - TO - 애플리케이션 방식)
    - RPC
    - MOM
    - ORB
    - OTM
    - DGE
