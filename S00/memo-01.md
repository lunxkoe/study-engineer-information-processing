# S01. 1과목 - 1 (소프트웨어 설계)

## 소프트웨어

### 소프트웨어의 특징
- 상품성
- 복잡성
- 변경 가능성
- 복제성

### 시스템의 개요와 기본 요소
- 기본 요소
    - 입력
    - 처리
    - 출력
    - 제어
    - 피드백

- 다이어그램
    - 입력 -> 처리 -> 출력
    - 제어 -> (입력, 처리, 출력)
    - 출력 -> 피드백

### 소프트웨어 위기
- 소프트웨어의 개발 속도가 하드웨어의 개발 속도를 따라가지 못함
- 원인
    - 하드웨어 비용을 초과하는 개발 비용의 증가
    - 개발 기간의 지연
    - 개발 인력 부족 및 인건비 상승
    - 성능 및 신뢰성 부족
    - 유지보수의 어려움에 따른 엄청난 비용

## 재공학

### 개념
- 개발의 생산성이 아닌
- 유지보수의 생산성으로 해결하려는 방법
- 사용자의 요구사항이나 기술적 설계의 변경 없이 프로그램을 개선하는 것

### 재공학의 장점, 목표, 과정
- 장점
    - 개발 시간 및 비용 감소
    - 품질 향상
    - 생산성 향상
    - 신회성 향상
    - 구축 방법에 대한 지식의 공유
    - 프로젝트 실패 위험 감소

- 목표
    - 유지보수성 향상이 최우선 목표
    - 복잡한 시스템을 다루는 방법 구현
    - 다른 뷰의 생성
    - 잃어버리 정보의 복구 및 제거

- 과정
    - 분석(Analysis)
    - 구성(Restructuring)
    - 역공학(Reverse Engineering)
    - 이식(Migration)

- 연관 깊은 유지보수 유형
    - Preventive Maintenance
    - 미리 예방

## 소프트웨어 설계 방법론

### 소프트웨어 생명주기
- 타당성 검토
- 개발 계획
- 요구사항 분석
- 설계 
- 구현
- 테스트
- 운용
- 유지보수

### 폭포수 모형(Waterfull Model)
- 선형 순차적 모델
- Boehm이 제시한 고전적인 생명주기
- 각 단계가 **순차적**으로 진행

- 단점
    - 중간에 요구사항이 변경되어도 적용할 수 없음

### 나선형 모형(Spiral Model)
- Boehm이 제시
- 반복적인 작업을 수행하는 점증적 생명주기 모형
- 점증적 모형, 집중적 모형
- **유지보수 과정이 필요 없음**

- 과정(아래 과정을 계속해서 반복하면 점증적으로 개발 고도화)
    - 계획 분석: 계획 및 초기 요구 분석
    - **위험 분석**: 1차 위험 분석
    - 개발 및 검증: 1차 프로토타입 개발(MVP)
    - 고객 평가: 고객 평가

### 하향식과 상향식 설계
- 하향식 설계 (위에서 아래)
    - 제일 상위에 있는 Main User Function에서 시작하여 기능을 하위 기능들로 나눠 가면서 설계하는 방식

- 상향식 설계 (아래에서 위)
    - 가장 기본적인 컴포넌트를 먼저 설계한 다음 이것을 사용하는 상위 수준의 컴포넌트를 설계하는 방식

### 프로토타입 모형의 개요
- 실제 개발될 시스템의 견본을 미리 만드는 것
- 개발이 완료되고 나서 사용을 하면 문제점을 알 수 있는 폭포수 모형의 단점을 보완
- 요구사항을 충실히 반영 가능

### HIPO(Hierarchy Input Process Output)
- 계층적 입력 처리 아웃풋
- 입력, 처리, 출력으로 구성되는 시스템 분석 및 설계와 시스템 문서화용 기법
- **가시적 도표**
- **총체적 다이어그램**
- **세부적 다이어그램**
- **하향식 소프트웨어 개발을 위한 문서화 도구** (폭포수형과 비슷함)
- 장점
    - 보기 쉽고 이해하기 숩고 유지보수가 쉬움

### V-모델
- 폭포수 모형에서 시스템 검증과 테스트 작업을 강조한 모델
- 요구사항 분석
    - 기능 명세 분석
        - 설계
            - 개발(코딩)
> 여기까지는 정적 테스트(코드 분석)

            - 단위 테스트
        - 통합 테스트
    - 시스템 테스트
- 인수 테스트
> 여기까지는 동적 테스트(코드 실행)
- **그림을 반드시 기억**

## XP(eXtremeProgramming)
- 빠르게 만들자

### 핵심 가치
- 소통
- 단순성
- 피드백
- 용기: 고객의 요구사항 변화에 능동적으로 대응
- 존중: 개발 팀원 간의 상호 존중을 기본으로 함

### XP Process
- Architectural Spike
    - 요구사항을 확인하기 위한 프로그램
- **User Stories**
    - 사용자의 요구사항
    - 표현(UML / Use Case와 목적이 같음)
- Release Planning
    - 몇 개의 스토리가 적용되어 부분적으로 기능이 완료된 제품을 제공하는 것으로 부분/전체 개발 완료 시점에 대한 일정을 수립
- **Iterator**
    - 하나의 릴리즈를 세분화한 단위이며 1~3주 단위로 진행
- Acceptance Test
    - 실제 테스트
- Samll Release
    - 릴리즈 단위를 기능별로 세분화

### XP 12가지 실천사항
- Fine Scale Feed-back
    - 페어 프로그래밍
    - 프랜닝 게임: 게임처럼
    - TDD
    - Whole Team: 고객을 프로젝트 팀원으로 상주

- Continuous Process
    - Continuous Integration: 상시 빌드 및 배포를 할 수 있는 상태로 유지
    - Design Improvement: 기능 변경 없이 중복성, 복잡성 제거
    - Small Release: 짧은 주기로 잦은 릴리즈를 함으로써 고객이 변경사항을 볼 수 있음

- Shared Understanding
    - Coding Standard
    - Collective Code Owenership
    - Simple Desing
    - System Metphor: 

- Programmer Welfare
    - Substainable Pace: 개발자에 충분한 휴식

### 효과적인 프로젝트 관리를 위한 3대 요소 (3P)
    - 사람(Person) - 인적 자원
    - 문제(Problem) - 문제 인식
    - 프로세스(Process) - 작업 계획

## Scrum

### 팀의 역할
- 제품 책임자
    - 개발 목표에 이해도가 높은 개발 의뢰자. **주로 사용자**
    - 제품 요구사항을 파악하여 기능 목록을 작성
    - 계획 수립까지만 임무를 수행
    - 스프린트가 시작되면 팀 운영에 관여하지 않음

- 스크럼 마스터
    - 업무를 배문 및 강요 X
    - 개발 과정의 장애 요소를 찾아 제거
    

- 스크럼 팀
    - 제품 책임자. 스크럼 마스터를 제외한 팀원
    - 개발자, 디자이너, 제품 검사기 등 모든 팀원

## 현행 시스템 분석

### 정의와 목적
- 회사 돌아가는 꼬라지 확인(**전체**)
- 현행 시스템의 제공 기능과 타 시스템과의 저보 교환 분석을 파악
- 소프트웨어, 하드웨어 파악

### 현행 시스템 파악 절차
- 1단계
    - 시스템(조직, 부서) 구성 파악
    - 시스템(조직, 부서) 기능 파악
    - 시스템 인터페이스 현황 파악

- 2단계
    - 아키텍처 파악(조직도)
    - 소프트웨어 구성 파악

- 3단계
    - 시스템 하드웨어 현황 파악
    - 네트워크 구성 파악

### 시스템(조직) 아키텍처
- 시스템 아키텍처 <-> 소프트웨어 아키텍처 (상호 관계)
- 소프트웨어 아키텍처 -> 소프트웨어 상세 설계

### 시스템 구성 파악
- 조직 내의 주요 업무를 기간 업무와 지원 업무로 구분하여 기술

### 시스템 기능 파악
- 시스템이 하는 일 별로 구분

### 인터페이스 현황 파악
- 각 조직이 어떻게 소통하는지
    - 데이터 형식: XML
    - 통신 규약: TCP

### EAI(Enterprice Application Integration)
- 기업 애플리케이션 통합(소프트웨어)
- 기업 내의 컴퓨터 애플리케이션들을 현대화하고, 통합하고 조정하는 것을 목표로 세운 계획

### 소프트웨어 구성 파악
- 내부에 있는 업무 처리 시스템 이름, 용도, 라이센스 등 파악
- 시스템 구축 시 많은 예산 비중을 차지하므로 라이선스 적용 방식과 보유한 라이선스 수량 파악이 중요

### 하드웨어 구성 파악
- 컴퓨터 사양
- 서버 이중화

### FEP
- 사실상 프론트엔드 데이터 입력 처리

### 플랫폼
- 종류
    - Java 플랫폼
    - .NET 플랫폼
    - IOS, Android, Windows, Linux
- 다양한 애플리케이션이 작동하는 기본이 되는 운영체제 소프트웨어를 의미함

### 플랫폼 성능 특성 분석
- 특성 분석 항목
    - 응답 시간
    - 가용성
    - 사용률

- 플랫폼 성능 특성 분석 방법
    - 기능 테스트: 부하 테스트
    - 사용자 인터뷰: 플랫폼 기능의 불편함을 인터뷰
    - 문서 점검: 유사한 플랫폼 기능 자료 분석

### 현행 시템 OS 분석 항목 침 고려사항
- 분석 항목: 현재 정보 시스템의 OS 종류와 버전, 패치 일자, 백업 주기 분석
- 고려사항
    - 가용성
    - 성능
    - 기술 지원
    - 주변 기기
    - 구축 비용(TCO) - Total Cost Ownership
        - HW, SW, 운영 교육, 기술 지원 등등의 전체적인 비용
- 메모리 누수

### 오픈소스 라이선스 종류
- GNU: GNU's Not UNIX
    - 모든 관련 정보를 돈으로 주고 구매하는 것을 반대하는 것이 기본 이념

- GNU GPLv1
    - 소스코드는 공개하지 않으면서 바이너리(실행)만 배포하는 것을 금지
    - 쉬운 소스 코드를 같이 배포해야함

- BSD
    - 아무나 개작할 수 있고, 수정한 것을 제안없이 배포할 수 있음
    - 단, 수정본의 재배포는 의무사항이 아님
    - 공개하지 않아도 되는 사용 소프트웨어도 사용해도 됨

- Apache 2.0
    - HADOOP
        - 다수의 저렴한 컴퓨터를 하나처럼 묶어 대량 데이터를 처리하는 기술

## 요구사항 개발

### 요구 공학(Requirement Engineering)
- 고객이 원하는 것을 어떻게 정확하게 뽑아낼 수 있을까를 고민하는 것
- 사용 기술
    - 자료 흐름도
    - 자료 사전
    - 더 구체적인 명세를 위해 소단위 명세서가 활용될 수 있음

### 목적
- 이해관계자 사이에 원할한 의사소통 수단을 제공함
- 요구사항 누락 방지, 상호 이해 오류 등 제거로 경제성을 제공
- 변경 이력 관리를 통하여 개발 비용 및 시간을 절약할 수 있음
- 비용과 일정에 대한 제약 서정과 타당성 조사, 요구 사항 정의 **문서화** 등을 수행

### 베이스라인(기준선)
- 명시적 합의 내용
- 조기에 요구사항을 명확히 확정

### SWEBOK에 따른 요구사항 개발 프로세스
- Software Engineering Body of Knowledge, 소프트웨어 공학 지식 체계

- 도출(Elicitation)
- 분석(Analysis)
- 명세(Specification) - 문서화
- 확인(Validation)

### 요구사항 도출
- 이해관계자가 식별됨
- **요구사항 도출 기법**
    - 고객의 발표
    - 문서 조사
    - 설문 업무 절차 및 양식 조사
    - 브레인 스토밍
    - 워크숍
    - 인터뷰
    - 관찰 및 모델의 프로토타이핑
    - Use Case
    - 벤치마킹
    - BPR(업무 재설계)
    - RFP(제안 요청서)

### 요구사항 분석
- 사용자의 요구사항을 도출해냄
- **요구 분석을 위한 기법**
    - 사용자 의견 청취
    - 사용자 인터뷰
    - 문서 분석과 중재
    - 관찰 및 모델링 작성 기술
    - 설문 조사

### 요구사항 분석 수행 단계
- 문제 인식
- 전개
- 평가와 종합
- 검토
- 문서화

### 요구사항 분류
- 기능 요구사항
- 비기능 요구사항

- 시스템 요구사항
- 사용자 요구사항

### 요구사항 분류 기준
- 기능 요구사항, 비기능 요구사항을 구분하고 우선순위 여부를 확인
- 요구사항이 하나 이상의 고수준 요구사항으로부터 유도된 것인지 확인
- 이해관계자나 다른 원천으로부터 직접 발생한 것인지 확인
- 요구사항이 제품에 관한 것인지 프로세스에 관한 것인지 확인하고 요구사항이 소프트웨어에 미치는 영향의 범위를 확인
- 요구사항이 소프트웨어 생명주기 동안에 변경이 발생하는지 파악

### 기능적 요구사항 vs 비기능적 요구사항
- 기능적 요구사항
    - 시스템이 실제로 어떻게 동작하는지 관점을 둔 요구사항

- 비기능적 요구사항
    - 성능, 보안, 품질, 안정성 등 실제 수행에 보조적인 요구사항

### 요구사항 명세 기법
- 문서화

- 정형 명세 기법
    - 수학적 기반/모델링 기반
    - **Z**, VDM
    - Petri-Net
    - 시스템 요구 특성을 정확하고 명세가 **간결하게 명세할 수 있음**
    - 명세/구현의 일치성
    - 낮은 이해도(수학적 이다보니깐,,)
    - 이해관계자의 부담 가중

- 비정형 명세
    - 상태/기능/객체 중심 명세 기법
    - 자연어 기반
    - SADT
    - 명세 작성 이해 용이
    - 의사전달 방법 다양성
    - 불충분한 명세 가능
    - 모호성

### 요구사항 명세 속성
- 정확성: 요구사항은 정확해야함
- 명확성: 단 한가지로만 해설
- 완전성: 모든 것이 표현(기능 + 비기능) 가능해야함
- 일관성: 요구사항 간 충돌이 없어야 함
- **수정 용이성**: 변경이 가능해야함

### 요구사항 확인
- 문서로 만들어진 내용을 확인하고 검증하는 단계
- **형상 관리**
- 도구의 필요성
    - 요구사항 변경으로 인한 비용 편익 분석
    - 변경 추적 가능
    - 요구사항 변경에 따른 영향 평가

### 형상 관리
- 개발 단계에서 도출되는 모든 것들(문서, 데이터, 자료, 프로그램)
- 버전 관리

### 요구사항 할당
- 아키텍처 구성 요소를 식별하는 활동

### 정형 분석
- 구문과 형식적으로 정의된 의미를 지닌 언어로 요구사항을 표현
- 정확하고 명확하게 표현하여 오해를 최소화
- 요구사항 분석의 마지막 단계

## 요구사항 확인 기법

### 요구사항 확인 기법의 종류
- 프로토타이핑
- 모델 검증
- 요구사항 검토
- 인수 테스트

### 프로토타이핑
- 시제품을 제작
- 도출되는 추가 요구사항을 지속해서 재작성하는 과정
- 요구사항 도출하기 위한 수단
- 절차:
    - 요구사항 분석 단계
    - 프로토타입 설계 단계
    - 프로토타입 개발 단계
    - 고객의 평가 단계
    - 프로토타입 정제 단계
    - 완제품 생산 단계

- 장점
    - 유용한 피드백을 제공
    - 의사소통이 좋아짐
    - 가변성이 급격히 감소함
    - 반복 제작을 통해 발전된 결과를 가져올 수 있음

- 단점
    - 프로토타입 수행 비용
    - 핵심 기능에서 멀어질 수 있음(사용자의 관심이 디자인 품질 문제로 집중 될 수 있음)

### 모델 검증
- 분석 단계에서 개발된 모델의 품질을 검증
- **정적 분석**:
    - 코드 확인
    - 문서 확인
    - 의사소통 경로를 검증

- **동적 분석**:
    - 직접 실행

### 인수 테스트
- 계약 인수 테스트
- 규정 인수 테스트
- 알파 검사
- 베타 검사
- 사용자 인수 테스트
- 운영 인수 테스트
